Hyperliquid · P2P sync gateway

hypersync
架构与节点同步流程

hypersync 是部署在 Hyperliquid 非验证节点前的多 peer、健康监控、自动故障转移的透明 P2P 网关。节点将 root_node_ips 指向 gateway；两个可独立开启、可叠加的标志在 transparent failover backbone 之上分别加装能力：--push 叠加多源 live block 加速 + active peer 故障转移，--cache 叠加 RAM 缓存冷启动。生产推荐同时开启 transparent + push：push 层任何降级都不影响节点继续从 transparent backbone 同步。

关键约束 · 协议 §4 §12 节点只会接收并应用自己主动拨出的 outbound peer 数据，入站推送不会被当作同步来源。因此 hypersync 必须写入节点的 root_node_ips，作为节点主动连接的 peer。这是整个网关方案成立的前提。
02

网关连接路由

HL 节点会连接 gateway 的 4000–4010 端口。gateway 根据端口和 4001 greeting 类型决定走透明转发、bootstrap 选择、缓存回放还是多源 live merge（入口：run_gateway）。--push 和 --cache 是两个正交标志：前者只影响 live 通道（是否叠加多源+故障转移），后者只影响 bootstrap 通道（是否命中 RAM 缓存）与 4002（cache 冷启动节点没有真实 peer 关系，才需要 fetch-forward）。

为什么所有连接钉同一个 active 4002 client-block RPC 如果落到一个没有和该节点建立过同步关系的 peer，可能返回 Peer-only request。因此 active 一旦由 4001 块流连接确定，其它通道会通过 dial_active 复用同一个上游。实现上也不会因为瞬时拨号失败立刻清空 active，避免连接抖动导致 RPC 被路由到错误 peer。

04

--cache 缓存冷启动 · 可选,独立于 --push

默认模式下，节点每次冷启动都要从真实 peer 流式拉取整份 state，重启越频繁越容易撞上该 peer 的 per-IP abci_state 限流。启用 --cache 后，gateway 会把完整 bootstrap 竞速抓取并缓存在 RAM 中；节点冷启动时直接从缓存回放，完全不请求真实 peer 的 state。代价：缓存冷启动的节点没有真实 peer 关系，其 4002 client-block catch-up 必须靠 gateway fetch-forward 代为请求，比 splice 直连更易受单个 peer 波动影响（见 §05 生产建议）。

05

--push 生产组合模式 · transparent backbone + 多源加速 + active 故障转移

生产环境的稳健做法是同时启用 transparent failover backbone 与 push 加速，而不是二选一。--push 把多源加速叠加在默认的 transparent bootstrap 之上：active peer 仍然是保底的 transparent backbone（转发 greeting、RPC、ack，并携带完整 live block 流），gateway 额外从 --live N（默认 5）个 peer 里挑 N-1 个 shadow peer 注入更快到达的区块副本，按 round 去重、fastest-first。关键设计：push/多源层任何降级都不影响节点继续同步——因为节点始终经由 transparent backbone 拿到完整数据，shadow 只是加速，不是必需路径。

4002 client-block RPC 在组合模式下走 dial_active → splice（不是 fetch-forward），因为节点经 transparent backbone 保有真实 peer 关系。这是组合模式比单独 --cache 更稳健的原因：4002 直连不受某个第三方 peer 限流/慢速影响，只受 active peer 本身的健康状况影响，而 active 本身有故障转移。

已验证 · 生产 node1 实测 组合模式下持续同步 5h15m（64 个采样，0 stall、0 真实错误、全程仅连网关）；额外做过 1 小时含节点重启的完整周期测试（transparent re-bootstrap → catch-up → 稳态，到 tip 后 0 新增真实错误）；并用主动断开网关到 active 的上行连接确定性验证了故障转移——节点在新连接上重连，网关 pin 到全新 active，同步在数十秒内恢复到实时速率。
已知限制 · pump_merge 无空闲超时 故障转移的触发条件是 active 连接报错关闭（EOF/RST）或 node 侧写失败，不是数据静默。如果 active peer 保持 TCP 连接打开但停止发送任何帧（既不断开也不给数据），当前实现不会检测到并触发切换——`pump_merge` 的 `read_exact` 调用没有包 `timeout()`，会无限期阻塞。这是已识别的差距，尚未修复；实测中未触发过（真实 peer 断连时通常是 RST/EOF，不是静默悬挂）。
06

proxy 模式故障转移状态机

proxy 模式使用固定 peer 列表，不依赖 peerd.sh 动态池。它通过显式健康监控完成 active peer 切换——这是与 §05 gateway --push 故障转移（撕除会话让节点重连）不同的另一套机制，二者互不依赖。

stateDiagram-v2
  [*] --> Active: 选定 active peer
  Active --> Active: 有数据流入(刷新 last_data_ms)
  Active --> Failover: 静默 > STALL_MS(20s)
或 dial 失败
  Failover --> Active: pick_next() 找到健康 peer

切换由 generation 计数器驱动：每次 failover 都会递增 generation，正在运行的透传协程检测到代次变化后主动断开，后续连接再路由到新的 active peer。

07

关键实现细节

下面这些实现点决定了 gateway 是否能稳定、低延迟地完成完整同步。

▸ 两种块序列化不可互换 （协议 §6，全链路最重要约束）

4001 上的 live/共识块（包含约 200KB 的 abci_block，round 在解压后 0x5e）和 4002 RANGE 响应里的 ClientBlock element（包含 commit_proof 的 child/grandchild QC，round 在 element 起点 +0x5d）是两种不同的 bincode 布局。ClientBlock 需要后续 round 才产生的 commit proof，不能从单个 live block 合成。因此 catch-up 必须向真实 peer 发起 4002 RPC，并将响应原样转发给节点——组合模式下这个 RPC 走 splice 直连 active，只有 --cache 冷启动的无 peer-关系节点才需要 fetch-forward。

▸ hotpath 优化 → ~135M frames/s

live merge 去重只需要读取 round。最新代码先走最快路径：如果 LZ4 block 起始 literal run 已覆盖前 0x63 个解压字节，就直接从压缩 payload 的 literal bytes 读取 0x5e 处的 round，用 get_unchecked（长度已校验）跳过边界检查，不分配解码缓冲、不复制字节；只有 literal 覆盖不足时才 fallback 到 lz4_first_n 的有界解码，最后才退回完整解压 oracle。10 轮 hotpath 埋点优化（Cargo 依赖 + main.rs 内 #[hotpath::measure] 属性宏，非独立文件）后为 31.6M → ~132-136M frames/s（约 4.3x，单次 bench 运行有 ±3% 波动），每轮均对照完整解压 oracle 验证 0 mismatch。

▸ lock-free round 去重

RoundDedup 不使用 mutex、hash set 或 ring-buffer。它把容量提升到 2 的幂，用 round & mask 定位 AtomicU32 slot，并通过一次 swap 判断该 round 是否已见过。serve 和 serve_push 当前使用 16,384 个 slot；极少数竞态最多导致重复转发一帧，节点端仍会按签名和状态去重。

▸ serve_push 的 greeting 顺序不变式

节点在 4001 上的第一次读取必须是 peer 的 greeting（≤1000 字节的控制帧）。serve_push 把 active 与多个 shadow peer 汇入同一条下行通道，若不加约束，某个 shadow 的第一个 live block 可能抢在 active 的 greeting 之前到达，导致节点把区块长度误判为 greeting 长度而报错断连。修复方式：在启动 shadow 注入器之前，同步读取并转发 active 的首帧（校验 len ≤ 1000）。这类协议时序问题只能靠抓包对比 len 分布验证，日志本身看不出竞态。

▸ I/O 与内存边界

所有 timeout(..).await 的读取都显式匹配 Ok(Ok(_))，避免把上游/节点提前关闭连接（Ok(Err(_))）误当成功继续、写出基于半截缓冲区的错误帧。对端可控的帧长度都有硬上限：bootstrap 单帧 ≤1.5GB（真实 abci_state 约 954MB），4002 响应 ≤64MB。is_routable 排除 0./127./10./192.168./169.254./255. 等常规私有/特殊段；对 172.* 段则只排除真正私有的 172.16.0.0/12（第二字节 16-31），不会把整个 172.0.0.0/8（包含 Cloudflare 等公网 172.64/13 段）误判为私有而丢弃。

4001、4002 都不是"一个端口一个接口"，而是按 greeting 位或请求内容复用了多种子协议（§8 协议文档）。下表对比每个子接口在 node1（真实 HL 节点，作为出站客户端 / 入站服务端）和 gw（当前部署的 gateway 子命令）上分别提供什么。
