Hyperliquid · P2P sync gateway

hypersync
架构与节点同步流程

hypersync 是部署在 Hyperliquid 非验证节点前的多 peer、健康监控、自动故障转移的透明 P2P 网关。节点将 root_node_ips 指向 gateway；gateway 负责上游 peer 池选择、透明转发、bootstrap 缓存、live block 多路合并和 failover，从而降低单个 peer 慢速、限流或断连对节点同步的影响。

01

系统架构

系统由四类组件组成：本地 HL 非验证节点、hypersync gateway、负责发现和探活的 peerd.sh，以及 Hyperliquid 主网的真实上游 peer 池。

关键约束 · 协议 §4 §12 节点只会接收并应用自己主动拨出的 outbound peer 数据，入站推送不会被当作同步来源。因此 hypersync 必须写入节点的 root_node_ips，作为节点主动连接的 peer。这是整个网关方案成立的前提。
02

网关连接路由

HL 节点会连接 gateway 的 4000–4010 端口。gateway 根据端口和 4001 greeting 类型决定走透明转发、bootstrap 选择、缓存回放还是多源 live merge（入口：run_gateway）。

为什么所有连接钉同一个 active 4002 client-block RPC 如果落到一个没有和该节点建立过同步关系的 peer，可能返回 Peer-only request。因此 active 一旦由 4001 块流连接确定，其它通道会通过 dial_active 复用同一个上游。实现上也不会因为瞬时拨号失败立刻清空 active，避免连接抖动导致 RPC 被路由到错误 peer。

04

--push 缓存冷启动

默认模式下，节点仍需从真实 peer 流式拉取整份 state。启用 --push 后，gateway 会把完整 bootstrap 缓存在 RAM 中；节点冷启动时直接从缓存回放，避免再次触发真实 peer 的 per-IP state 限流。

05

故障转移状态机

proxy 模式使用固定 peer 列表，不依赖 peerd.sh 动态池。它通过显式健康监控完成 active peer 切换。

state · proxy-failover
stateDiagram-v2
  [*] --> Active: 选定 active peer
  Active --> Active: 有数据流入(刷新 last_data_ms)
  Active --> Failover: 静默 > STALL_MS(20s)
或 dial 失败
  Failover --> Active: pick_next() 找到健康 peer
旧 peer 进 cooldown(30s)
  Failover --> Active: 无健康替代
→ 保持当前(避免频繁切换)

切换由 generation 计数器驱动：每次 failover 都会递增 generation，正在运行的透传协程检测到代次变化后主动断开，后续连接再路由到新的 active peer。

06

关键实现细节

下面三点决定了 gateway 是否能稳定、低延迟地完成完整同步。

▸ 两种块序列化不可互换 （协议 §6，全链路最重要约束）

4001 上的 live/共识块（包含约 200KB 的 abci_block，round 在解压后 0x5e）和 4002 RANGE 响应里的 ClientBlock element（包含 commit_proof 的 child/grandchild QC，round 在 element 起点 +0x5d）是两种不同的 bincode 布局。ClientBlock 需要后续 round 才产生的 commit proof，不能从单个 live block 合成。因此 catch-up 必须向真实 peer 发起 4002 RPC，并将响应原样转发给节点。

▸ 热路径优化 → 795×

live merge 去重只需要读取 round，因此 lz4_first_n 做的是有界 LZ4 解码：解出前约 99 字节（0x63）即可停止，不需要解完整个约 200KB 的 block。当前 RoundDedup 使用 round % cap 的 slot window 做常数时间去重；README 中的 bench 显示该热路径相对完整解压有数量级优势。

▸ 去重窗口的边界

去重 key 是 u32 round，gateway 的 live merge 窗口容量为 131072。该窗口面向近实时顺序增长的 live block 流设计；如果上游异常回放很旧的 round，窗口外数据可能被视为新块，需要依赖节点端签名和状态校验兜底。

07

运行模式速查

子命令作用
gateway <peers-file> [--push] [--live N] [--retain N]完整 P2P 网关（推荐）。使用 peerd 动态池，提供透明转发、bootstrap 缓存、多路 live 合并和 4002 RPC 转发。
proxy <peer1,peer2,…> [--push]固定 peer 列表的透明代理，覆盖 4000-4010，并在上游 stall 或拨号失败时切换 active peer。
relay <host>单上游多端口透明中继，用于最小化转发路径或本地调试。
serve <port> <peers>多上游 live block 合并服务，按 round 去重并转发最先到达的副本。
cache <port> <upstream>从单上游捕获 abci_state 并缓存，以本地速度服务后续连接的节点。
mock / bench回放已捕获 block 进行测试，或运行 round 提取与去重热路径基准。
基于闭源 hl-visor / hl-node 的明文 TCP P2P 行为逆向整理 · 块由节点端到端验签，gateway 只做字节级转发与路由
