Hyperliquid · P2P 同步网关

hypersync
架构与节点同步流程

hypersync 是放在 Hyperliquid 非验证节点前面的透明 P2P 同步网关。节点只连 gateway；gateway 替它挑一个真实 peer 做主连接，并在需要低延迟时额外连几个 peer 抢最快的 live block。先记住一句话：active 是主连接，shadow 是加速连接，4002 是追块 RPC。

active peer（主上游）

当前会话真正依赖的上游。启动快照、4002 追块、ack 和普通 gossip 都优先走它，节点也靠它维持真实 peer 关系。

shadow peer（旁路加速源）

--push 下额外连接的 peer。它只拿 live block，谁先拿到某个 round 就先转发；重复 round 会被丢掉。

主链路

最基础的透明转发链路：node ↔ gateway ↔ active。它让节点像直接连真实 peer 一样工作。

--cache 与 4002

--cache 只缓存冷启动的大启动快照。用缓存启动后，节点没有真实 peer 关系，所以 4002 需要 gateway 代请求再原样转回。

02

为什么所有连接钉同一个 active 4002 不是随便找个 peer 就能问。某些请求要求对方认识这个节点；如果打到陌生 peer，可能返回 Peer-only request。所以 4001 一旦选出 active，4002 和其它通道就尽量复用这个 active。

04

--cache 缓存冷启动 · 避免重复拉状态

--cache 解决的是一个具体问题：节点重启时不想再从真实 peer 拉一遍约 4.5GB 的启动快照。gateway 先在后台抓一份完整启动快照放进 RAM；节点冷启动时，gateway 直接把这份缓存回放给它。代价也很明确：节点这次不是从真实 peer 拿状态，所以后续 4002 追块不能稳定直连 active，只能由 gateway 代问真实 peer。

05

--push 生产推荐模式 · 主连接保底,旁路抢最快块

--push 不会替换透明转发，它是在透明转发上加速。active peer 仍然是主连接，负责完整数据和 4002；shadow peer 只是额外旁路，负责抢更快的 live block。shadow 断开最多影响加速效果；active 断开才会让节点重连并重新选择 active。

角色它做什么异常时会怎样
active主上游。转发 greeting、4002、ack、普通帧和 live block，是节点真实 peer 关系的来源。关闭当前会话，节点重连，gateway 选新的 active。
shadow旁路加速源。只请求 send_abci:false live block，拿最快到达的 round。后台重连；同步不应该中断，只是少一个加速源。
dedup按 round 去重。active 和 shadow 谁先到就转发，后到的同 round 丢弃。少量重复也由节点端验签和状态检查兜底。

这也是为什么推荐 gateway --push 作为生产默认：它既保留“节点真的连到了一个 peer”的关系，又从多个 peer 抢最快 block。4002 在这里仍然直连 active，不走 --cache 的代请求路径。

已验证 · 生产节点实测 组合模式下持续同步 5h15m（64 个采样，0 次卡住、0 个真实错误、全程只连网关）。另外做过 1 小时含节点重启测试：重新冷启动、4002 追块、进入稳态都正常。主动断开 active 上行连接后，节点会重连，gateway 会选新 active，同步在数十秒内恢复。
已知限制 · pump_merge 无空闲超时 故障转移现在靠“连接报错/关闭”触发，不靠“长时间没数据”触发。如果 active TCP 连接一直开着但完全不发数据，当前实现不会主动切换。这是代码边界，不要把它理解成完整健康检查。
06

proxy 模式故障转移状态机

proxy 是另一种运行方式：不用 peerd.sh 动态池，而是你手动给一组固定 peer。它会监控当前 active 是否长时间没数据或拨号失败，然后切到下一个 peer。注意：这套机制和 §05 的 gateway --push 不是一回事。

状态机 · proxy 如何切换 active
stateDiagram-v2
  [*] --> Active: 选定 active peer
  Active --> Active: 有数据流入(刷新 last_data_ms)
  Active --> Failover: 静默 > STALL_MS(20s)
或 dial 失败
  Failover --> Active: pick_next() 找到健康 peer
旧 peer 进 cooldown(30s)
  Failover --> Active: 无健康替代
→ 保持当前(避免频繁切换)

实现上用 generation 做版本号：每次切换 active 都加一，旧连接发现版本变了就主动断开，后续新连接会走新的 active。

07

关键实现细节

这一节只保留几个必须知道的技术边界。看不懂代码也没关系，重点是知道为什么不能随便改这些逻辑。

▸ 4001 的 block 不能拿来回答 4002

4001 live block 和 4002 client-block 响应不是同一种数据格式。4002 里的 ClientBlock 带有后续 round 才能形成的 commit proof，不能从单个 live block 现场拼出来。所以追块必须真的去问 4002，再把响应原样给节点。默认和 --push 模式是直连 active；只有 --cache 冷启动后没有真实 peer 关系，才需要 gateway 代请求。

▸ 为什么只读 round 就够了

多源合并只需要判断“这个 round 是否已经转发过”，不需要完整解压每个 block。代码优先从 LZ4 开头的字面量里直接读 round；不满足条件时才做小范围解码，最后才退回完整解压校验。这样热路径才能跑到约 132-136M frames/s。

▸ round 去重怎么做

RoundDedup 用一个固定大小的原子数组记录最近见过的 round。新 block 到来时，用 round 定位一个 slot，原子替换并判断旧值是否相同。相同就是重复，丢掉；不同就是新 round，转发。极少数并发竞态最多多转一次，节点端还会验签和按状态去重。

▸ 为什么 shadow 不能抢在 greeting 前面

节点刚连上 4001 时，第一次读到的必须是 active peer 的 greeting。如果 shadow 的 live block 先到，节点会把 block 长度误当成 greeting 长度，然后直接断连。所以 serve_push 必须先读并转发 active 的首帧，再启动 shadow 注入。

▸ I/O 与内存边界

所有 timeout(..).await 的读取都显式匹配 Ok(Ok(_))，避免把上游/节点提前关闭连接（Ok(Err(_))）误当成功继续、写出基于半截缓冲区的错误帧。对端可控的帧长度都有硬上限：bootstrap 单帧 ≤1.5GB（真实 abci_state 约 954MB），4002 响应 ≤64MB。is_routable 排除 0./127./10./192.168./169.254./255. 等常规私有/特殊段；对 172.* 段则只排除真正私有的 172.16.0.0/12（第二字节 16-31），不会把整个 172.0.0.0/8（包含 Cloudflare 等公网 172.64/13 段）误判为私有而丢弃。

09

端口接口对比：4001 / 4002 / 其它

最容易混的是 4001 和 4002。简单说：4001 管状态和 live block；4002 管追块、查高度、查 peer。gateway 对 4001 会做缓存或多源合并；对 4002 通常只是转发，只有 --cache 冷启动后才代请求。
