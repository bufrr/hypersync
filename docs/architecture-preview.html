3. 协议边界

HL P2P 是 plaintext TCP。gateway 主要处理 frame 转发、greeting 判断、active peer 绑定，以及 live merge 时的 round 提取和去重。

Frame

u32 BE length + 1 byte type + payload。type=0 是 control，type=1 是 data。

Greeting

send_abci=true 请求完整 snapshot；send_abci=false 只请求 live blocks。

Round extract

live block 是 lz4_flex，round 在解压后的 0x5e；代码只解前约 99 字节。

关键限制：4001 live block 和 4002 ClientBlock response element 是两种不同 bincode 序列化。 live block 缺少后续 commit proof，不能转换成 ClientBlock，gateway 必须向真实 peer 请求 4002 response 并原样转发。

6. `gateway --push` 缓存与多源 live merge

`--push` 下，gateway 会后台捕获完整 bootstrap 原始字节流，节点冷启动时可以从 RAM replay。 live/resume 阶段则保持 active peer 透明连接，并额外连接 shadow peers 抢最快 live block。

Bootstrap cache

  • 后台任务向 top pool peers 发送 send_abci=true。
  • 捕获 abci_state + EVM KVs + tail bytes，按 frame 原样保存。
  • 900 到 970 MB 附近要求速度超过 10 MB/s，避免缓存慢 peer。
  • bulk 传输速率持续跌到 live trickle 后停止捕获。
  • 节点冷启动时直接 replay boot_blob，不消耗真实 peer 的 state quota。

Live merge

  • active peer 连接保持透明，承载节点 outbound/control/state 相关字节。
  • shadow peers 使用 send_abci=false，只拉 live blocks。
  • pump_merge 从 type=1 payload 提取 round。
  • RoundDedup 只放行每个 round 最先到达的一份。
  • 非 active peer 的大 frame 会被 drain，不转发给节点。
当前代码行为：serve_client_blocks 的 _buf 参数没有被使用。 4002 catch-up 会把节点 request 转发给 active/pool peer，并把 response 原样返回，而不是从本地 live buffer 回答。

8. 同步完成信号

同步完成不是单个网络事件，而是一组节点侧状态变化。

Bootstrap 完成

节点 certify/commit snapshot，并写出 hyperliquid_data/visor_abci_state.json。

Catch-up 完成

4002 client-block range query 追到 committed tip 附近。超前时可能短暂收到 client block round too large。

Steady state

4001 live blocks 持续应用，日志出现类似 new app hashes reached quorum。
